1. Tethers
I want to be able to simulate tethers/ropes/cables between crafts.
Can the same part object belong to multiple crafts?
If I extend part to have another parent reference, then can it push forces to two parents?
Only one parent has to update it each loop, so the second craft might not need a reference to the part. 
It just needs to be able to push separate forces to each craft. 
Following the framework, inter-craft interactions are supposed to go through fields, but this seems like a special case. 

2. Ocean Surface
Right now it's unclear how I will simulate the surface of an ocean. 
A normal FluidField where I can query the state (density, velocity) of the fluid works great when it's relatively continuous. 
However the earth has atmosphere and ocean with a defined boundary between them. 
To simulate boats and other floating objects, each part will query the fluid state and be buoyant if in water and not buoyant in air.
If parts only sample one point, they will oscillate about the surface of the ocean. 
My best solution is to have each buoyant part sample a few different points. 
For example, a simple cylindrical hull can sample 4 points along the gravity vector. 
This should make the behavior a bit more continuous - instead of being fully in/out of the water it will have some gradual force dropoff.

I could also create a Field called "SeaSurface" that exposes a "heightAboveSeaLevel" method or something that given a position returns its height above sea level.
It would be synchronized with the FluidField and allow certain part models like Hulls etc. to accurately get their bouyant force. 
Instead of having to synchronize this with the FluidField, I'm leaning towards just folding it into a subclass. 
I'd create subclass of FluidField called "OceanAtmosField" that serves this "heightAboveSeaLevel" as well as the normal fluid state.

3. Part models are field-dependent
Following from the previous point, it would be nice if Part models could be dependent on which Fields are included at compile time. 
For example a "Hull" part absolutely requires a FluidField, and will default to the sampled points algorithm to compute buoyant force.  
However if the "OceanAtmosField" is also present, it will augment that with a calculation that uses the smoothly computed depth. 

4. Visualizer
I want some way to visualize my simulations in something like rerun.io. 
I want to easily switch between camera angles following different craft frames, scene frames, planet frames, etc. 
I want to be able to visualize the wrenches that certain parts are generating as arrows.
I want crafts/parts to support custom meshes. 
It would also be nice if the user could click on crafts and parts within the rendered simulation to select them,
to see further information that is passed up to various dashboards.  

Obviously, there should be nothing visualization-related in the core C++ library, but 
I think it makes sense to add optional visualization information to the Python part/craft descriptors.
Then the visualizer can use the python codegen to know how to render various crafts. 

I also want custom things to be rendered for certain parts. For example in the "Camera" part python descriptor, 
it should describe how to render the view frustum. This keeps the C++ part model completely visualization-agnostic, but 
also makes it easy for the user to visualize their simulation without having to write all that themselves. 

Or in the "Thruster" python part descriptor, the user can write a small bit of code that specifies the dashboard that shows up
-when the part is selected in the visualizer. 

So the base python part descriptor will have a variety of default visualization functionality. 
(render the part as a small box/coordinate axis), show the part name and type when clicked on, toggle to show the wrenches generated by the part. 

5. Workflow Flexibility
As I was thinking about adding visualizer info to the python descriptors, I realized the python codegen has a lot of potential to make simulation-only workflows very seamless.  
Currently this library can only be used if built with a user-written C++ main, 
but it might also be nice to make the python codegen capable of generating a basic C++ main that simply serves the desired simulation objects over zenoh. 
The python part descriptors can optionally specify how the Part models will be read from/written to. 
For example, if the user wants to send the fake IMU data to a different process, instead of hooking that up themselves by writing the C++, 
they use the python descriptor to specify a zenoh port that the data will be exposed over and the C++ is automatically generated. 
In theory, simulation-only users won't have to touch any C++. They configure with python, it generates the c++, then they build and run the executable. 
All of the information they are interested in is sent over zenoh, so they can even write simple controls in python to control their simulated craft over zenoh. 
In the future, I could even expand this to support ROS2 integration (instead of zenoh the python speficies some ros topics), but I'm not worrying about that now.
