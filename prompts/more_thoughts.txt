
There are a few main workflows I envision this library supporting:
1. Pure simulation with external controls:
The user writes python to specify their craft and generate the simulation executable. Then runs the executable and uses Zenoh to read and control their crafts. The python part descriptors include visualizer information, meaning the user doesn’t have to worry about instrumenting the simulation to make the visualization. The user can still do custom visualizations by hooking into the Zenoh outputs generated by manta. 
2. Pure simulation with internal controls:
The user writes C++ that includes the manta library and builds their own executable. They can still use the python codegen to write their Craft specification (it is recommended), but it is technically optional. Provides more flexibility to the user regarding how the simulation is run. Can still use Zenoh to work with external processes. 
These first two workflows also support hardware in the loop. A user can choose to pipe real sensor data over Zenoh into the simulation. For example there will be a “MotionTrack” Part that can be then added to a Craft and will exert the required forces to make that Craft follow the input trajectory. This can be used to generate new simulated sensor streams that didn’t previously exist. For example if the user wants to add a simulated camera to their drone, they play back the data collected from their real flight and the simulation fills in the rest. It also works in reverse. Users can pipe simulated data out over Zenoh into real actuators then instrument those actuators and feed that data back to the simulated actuators. 

3. Onboard state estimation and control:
The user optionally writes python codegen to specify the craft (same as option 2), then they write C++ to do their state estimation and controls, leveraging manta’s model and estimator framework. Zenoh is entirely unnecessary here, except for if the Craft model is interacting with other Craft models over a manta::Field. For example a drone swarm communicating over some physical layer: there will be a separate process serializing from the physical layer into Zenoh, then manta takes in the Zenoh so each drone has knowledge of how the others will affect its dynamics. 

When I was conceptualizing this simulation library I wanted to take full advantage of the fact that it's automatically differentiable and allow for model optimization/system identification by feeding in real data and using an optimizer to make the simulated match real. For instance, if I had ground truth motion capture data of my drone flying, I could use it to automatically fine-tune the model by tweaking part parameters. Does the current architecture support this in future? I know ceres has some optimization functionality that might be good for the low numbers of parameters. Given that the craft construction and part model values happen at compile (specified via codegen), would it be possible to make them runtime variables?

Would it also be possible to create Python bindings for this library so I could use something like pytorch to optimize? Python is also easier for most people, and is more than fast enough for boats/submarines.
