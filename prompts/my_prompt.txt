High Level Feature Goals - What is Manta trying to accomplish and innovate on?

1. Modular simulation:
Large numbers of Crafts of a variety of craft types (boats, satellites, planes, all in the same simulation.) This is accomplished by the Part and Field models. Users can add multiple Crafts within the same process, and when this becomes too slow, split Crafts across multiple processes that don’t have to run on the same computer. The Fields preserve the same interface, except now exchange information over Zenoh. I’m not sure how this architecture will work, will each process have a duplicate Field that synchronizes over Zenoh? Or will there be a “master” Field that all the other processes query and perturb? This ties into feature 4. 

2. State estimation and controls for a variety of crafts:
This is accomplished by using autodiff and leveraging the existing modular craft modeling system. The user can get state of the art EKF/UKF in a few lines after specifying their craft. The compiled C++ makes this fast and potentially portable to embedded targets. 

3. Model optimization and system ID:
Users can compare real-data vs simulated data and optimize the craft models to be more accurate. 

4. Networked system models for swarming:
When using manta for state estimation on a real robot, each robot can add a separate craft’s model to its current simulation/model world. For example, a networked swarm of drones can communicate their positions, then each one adds the others to its own manta world model, allowing them to predict interactions between each other's propwash, etc. Another example would be a swarm of undersea autonomous robots. When one robot registers an acoustic disturbance, it automatically pushes it to the shared field model, which means that when another robot in the swarm also perceives it, its system model will agree with the presence of the disturbance and boost the kalman gain/confidence in that sensor's reading. So by networking multiple robots, you can effectively model the entire swarm, which buys you some accuracy by virtue of more sensor data (law of large numbers). It’s supposed to be an easy benefit of the manta framework, not requiring lots of extra code to implement. 

5. Easy interface + Full control if wanted
Allowing users to interact with the C++ library, or simply call Python codegen accomplishes this. 

With these high level goals fleshed out, I will delve into some more technical topics of discussion. 

How to handle planets:
In the original manta plan.md, I described the frame hierarchy as world -> planet -> scene -> craft -> part. However it’s unclear how planets are added to the simulation. One thing I know for sure is that I don’t really want planets to be necessary for simulating. Users should be able to play with crafts in a uniform fluid/vacuum without worrying about any planets. Users should also be able to add many planets to their world and create custom solar systems. Maybe this means creating a default “Void” Planet that is stationary with no gravity. Modeling Planets is more than just enabling the user to do space missions across the solar system. It’s a key component of accurate physics. I want to be able to simulate the coriolis effect, etc. (for inertial navigation and AHRS to work correctly). I was thinking of making a dedicated Planet object that acts as a “composite Field” of sorts. It defines the planet frame (can be rotating/moving relative to the world frame) and adds disturbances to a bunch of Fields. So for example the Earth : Planet class would define the world frame as rotating at ~7.28 rad/s and it requires the FluidField, GravityField, MagField Fields, (similar to how a Part model requires certain Fields) and adds the ocean and atmosphere as disturbances to FluidField, adds its gravity map to the GravityField, and its magnetic map to the MagField. This way I would remove the OceanAtmosField and treat FluidField as a continuous Field that extends throughout the entire sim world. Planets add “disturbances” to create bubbles of ocean and atmosphere that are fixed to their planet frame. The “heightAboveSeaSurface()” method can be handled by a separate SeaSurface Field that the Earth class adds a disturbance to. So the Hull part model would just switch functionality based on inclusion of the SeaSurface Field. The GPS part model will require the Earth planet, in the same way other Parts can require certain Fields to be compiled. What do you think of this idea?

Field and Part Naming:
Does it make sense to call the Field class “Field”? The name makes sense, in that they are representing fields with excitations (disturbances) but in the context of software, it might be annoying, as function arguments are commonly referred to as fields. “Part” is similarly ambiguous. Changing to MantaPart and MantaField is ugly and I think both should already be in the manta namespace anyway like manta::Part, manta::Field. Do you think the names are bad? Do you have some alternative suggestions?

Frame conventions:
I originally envisioned the framed kinematic state queries to work as follows:
part.position<WorldFrame> - returns position of the Part’s origin in the world frame
part.position<SceneFrame> - returns position of the Part’s origin in the scene frame
part.position<CraftFrame> - returns position of the Part’s origin in the craft frame
But we get to part.position<PartFrame> and it seems odd to have it just return 0. By this convention, part.anything<PartFrame> would be 0, as the part frame doesn’t move relative to itself. However, you changed part.velocity<PartFrame> to mean the world-frame velocity as expressed in the part frame (body-frame). I don’t agree with this change, as it creates a contradiction in the convention, but I acknowledge that perhaps we need to revisit the convention. You’re right that it should be easy for the user to get the body-frame world velocity, acceleration, etc. Sticking to the original convention, the user would call part.velocity<WorldFrame> then rotate it by the inverse of part.orientation<WorldFrame> to get what is now returned by part.velocity<PartFrame>
How can we revamp this to be consistent AND seamless. 

Testing articulating parts:
I came up with a simple test case to verify that the CompositePart joint dynamics are working as expected. Create a craft and add a mass part offset from the origin by (+r, 0, 0) with a certain mass and moi. Then add a Motor CompositePart (articulating joint). Then on the motor, add an identical part of the same mass and moi but offset by (-r, 0, 0). So the craft (root part) has two children, Part1 and the Motor. The Motor has one child, Part2, where Part1 and Part2 have identical parameters and opposite offsets relative to the motor shaft axis (such that they span 180 degrees). Now, spinning the motor shouldn’t change the shared COM of the Craft. Imagine flexing a hinge, where each half of the hinge is one of our parts. If the hinge was floating in space with nothing to push on, the total center of mass of the system should stay put.
